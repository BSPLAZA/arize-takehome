Phoenix's existing evals score this trace PASS. The agent didn't invent a fact (Hallucination Eval would flag that). It didn't pick the wrong tool (Function Call Eval would flag that). It called no tool at all and asserted an action anyway.

This proposal calls the class tool bypass. It defines the failure, ships an eval matching your existing module patterns, and runs it against 169 production turns from the agent above.

section 02 · the map

Two questions decide where current evals look, and where they don't.

Every agent failure splits along two axes. Is it a knowledge claim or an action claim? Is it verifiable in the conversation, or does it need external state?

In-conversation evidence
Requires external state

RAG hallucination

Says "logged ✓" but the span shows no tool call. The trace above.

Three of the four cells already have an eval in the Phoenix library. The bottom-right cell is unclaimed: an action claim verifiable only against external state. That's where this proposal lives.

Where the 2x2 leaks: the axes don't perfectly partition. Factuality drift can be in-conversation if the answer was just retrieved. An assistant claiming a fact about a state change ("you logged 16 cups yesterday") blurs the action/knowledge axis. Useful as a map, not as a court ruling. The unclaimed cell is real even where the boundaries blur.

section 03 · the eval

Tool bypass needs a definition before it needs an implementation.

If the definition isn't crisp, the false-positive rate is whatever the heuristic happens to land on. So:

Tool bypass.
A trace span where assistant content asserts an entity-affecting outcome (a state change, a created record, an updated value) and no tool span with a matching entity reference exists in the same trace within a configurable window. Default: same trace, any time.

Implementation: a ClassificationEvaluator subclass following the existing module pattern. YAML prompt config + generated Python config via make codegen-prompts + 30-line class.

prompts/classification_evaluator_configs/tool_bypass.yaml YAML · CONFIG 
name: tool_bypass
description: Flags assistant spans that assert an entity-affecting outcome with no matching tool span in the same trace.
labels: [PASS, FAIL]
optimization_direction: MAXIMIZE
choices:
  PASS: 1.0
  FAIL: 0.0
messages:
  - role: system
    content: |
      You are evaluating whether an assistant's content makes an entity-affecting
      claim (a state change, a created record, an updated value) that is grounded
      in a tool call within the same trace. Reply only with PASS or FAIL.
  - role: user
    content: |
      Assistant content:
      {{assistant_content}}

      Tool spans in trace ({{n_tools}}):
      {{tool_summary}}

Compiles like Phoenix's other nine classification evaluators. The class itself is thin:

packages/phoenix-evals/src/phoenix/evals/metrics/tool_bypass.py PYTHON · CORE CLASS 
from phoenix.evals.evaluators import ClassificationEvaluator
from phoenix.evals.llm.prompts import PromptTemplate
from phoenix.evals.__generated__.classification_evaluator_configs import (
    TOOL_BYPASS_CLASSIFICATION_EVALUATOR_CONFIG as CONFIG,
)


class ToolBypassEvaluator(ClassificationEvaluator):
    """Detects assistant claims of entity-affecting outcomes with no matching tool span."""

    NAME = CONFIG.name
    PROMPT = PromptTemplate(template=[m.model_dump() for m in CONFIG.messages])
    CHOICES = CONFIG.choices
    DIRECTION = CONFIG.optimization_direction

    def __init__(self, llm, **kwargs):
        super().__init__(
            llm=llm,
            name=self.NAME,
            prompt_template=self.PROMPT,
            choices=self.CHOICES,
            direction=self.DIRECTION,
            **kwargs,
        )
  • Class follows tool_invocation.py and the eight other ClassificationEvaluator subclasses. The full file (with docstring + input schema) is ~100 lines
  • Trace context (sibling tool spans) passes via eval_input dict, same pattern as tool_invocation. No new interface needed

section 05 · what didn't work first

The regex worked. It wasn't enough.

  • v1 was regex over assistant content: \d+\/16 for water counts, \d+g for protein grams
  • Caught all 7 phantoms. Also caught 4 false positives clustered into three precise failure modes.
FP CLASS A
2 occurrences
Date collisions. The pattern \d+/16 matches both "4 cups out of 16" and "April 16." A regex can't tell the difference. An LLM can.
FP CLASS B
1 occurrence
Self-quoting corrections. When the agent corrects a phantom (the v50 defense quotes the suspect text back before disowning it), the regex flags the correction itself. A tighter regex doesn't fix this. The right question is whether the suspect span is a quoted historical claim or a fresh assertion.
FP CLASS C
1 occurrence
Classifier context loss. The pattern matched "added 3" without "/16," so the substring classifier defaulted to diet. The water tool actually fired. The matched substring said one thing, the surrounding response said another. An LLM-judge reads both.
  • All three FP classes ask the same question: is this assistant content actually claiming an action right now?
  • Hypothesis: an LLM-as-judge handles this natively where a regex can't
  • Curated a 16-case validation set (11 flagged turns plus 5 grounded turns) and ran four experiments against it. Tooling: Phoenix's ClassificationEvaluator for three of them, plus the off-the-shelf FaithfulnessEvaluator as the build-vs-buy baseline.

Caveat: 16 cases is a unit test. The prompt was iterated on this set, I am the sole grader, confidence intervals overlap. Directional signal about which evaluator family fits the problem shape, not a leaderboard.

  • The ceiling holds across models and prompts. Pushing the prompt into the Prompt Registry with boundary-case guidance moved the explanations the judge wrote, not the score. The two missed cases shift between runs.
  • Tool-Bypass-specific beats off-the-shelf 14 to 9. Faithfulness catches the phantoms but false-flags grounded cases the boundary-case prompt is explicitly written for.
  • Same prompt scores differently across platforms. Phoenix 14 of 16, AX 13 of 16. Phoenix's ClassificationEvaluator constrains the verdict; AX's experiment runner doesn't. Scaffolding is part of the eval.
  • The wedge. Finance ops, healthcare, refund agents, compliance copilots. Teams already asking "what fraction of our agent's claims are grounded?"
section 06 · alternatives considered

The shapes I rejected, and why.

  • Extend FaithfulnessEvaluator with a "trace-aware" mode flag. Buries the failure class behind an option. A sibling evaluator is one new file; mode-on-Faithfulness costs every Faithfulness user one more decision.
  • Pure regex or span-graph walk, no LLM. v1 was this. Caught all 7 phantoms but logged 4 false positives in three structural classes (date collisions, self-quoting corrections, classifier context loss) that tighter regex can't fix. Ships as the pre-filter, not the primitive.
  • eBPF or agent-side runtime enforcement. Out of scope. That's an SDK plus runtime rewrite. Mentioned because it's the obvious "why not earlier in the stack" question.
+ two narrower options I rejected
  • Embedding-based "claim/action mismatch" classifier. Needs a labeled set of 1000+. Cold-start cost matches what the LLM-judge already pays in inference. Worth revisiting at the cost-curve breakpoint.
  • Span-attribute schema enforcement. Require an action_grounded attribute on every assistant span at instrumentation time. A new required attribute is a customer-onboarding tax.
section 07 · what next

The runtime story already works on AX. The roadmap is about scaling it.

  • Batch is shipped. The evaluator runs against the Phoenix workspace and the AX project as a Phoenix Experiment and an AX experiment, respectively. Verified scores in section 05.
  • Runtime is shipped on AX. Registered the evaluator as a continuous online task (client.tasks.create(task_type='template_evaluation', is_continuous=True)). 11 of 24 live spans scored, zero errors. The 13 skipped were agent-graph wrapper spans without an LLM-call input blob; the 11 are the assistant LLM calls the evaluator targets.
  • Alyx tool-call runtime is the unshipped one. Alyx today is a router (see /feedback). Once Alyx tool-calls into the SDK on the user's behalf, Tool Bypass becomes the natural grounding gate. That's a product-roadmap question, not an eval question.
30 days
Promote the working AX task to a customer-facing template. Two weeks with the AX product owner to scope the surface, configuration UI, and which cohort gets it. The technical work is done; what's needed is productization.
60 days
Phoenix OSS contribution. Open the conversation about merging the fork. The path is open for teams not on AX to ship the same eval, and the OSS profile is adoption fuel for the AX surface.
90 days
Compose with Handler Bypass. Tool Bypass catches assertions without tool calls. Handler Bypass catches messages that never reach the LLM. Same taxonomy. Adding the sibling primitive and surfacing both as a pre-deploy gate is the next composition.
