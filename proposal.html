the proposal

Tool Bypass.

An eval primitive for Phoenix that catches a failure mode the existing library misses. I defined it, wrote the LLM-as-judge version in your repo's style, ran the regex variant against 169 historical DM turns from one of my agents (7 real phantoms, 4 false positives worth keeping), and re-ran it live against 57 post-defense spans flowing into Phoenix Cloud right now (zero phantoms, μ 1.00). The eval flags the bug where it exists. It stays quiet when defenses hold.

Phoenix's existing evals score this trace PASS. The agent didn't invent a fact (Hallucination Eval would flag that). It didn't pick the wrong tool (Function Call Eval would flag that). It called no tool at all and asserted an action anyway.

This proposal calls the class tool bypass. It defines the failure, ships an eval matching your existing module patterns, and runs it against 169 production turns from the agent above.

section 02 · the map

Two questions decide where current evals look, and where they don't.

Every agent failure splits along two axes. Is it a knowledge claim or an action claim? Is it verifiable in the conversation, or does it need external state?

In-conversation evidence
Requires external state
Knowledge claim
RAG hallucination
✓ Phoenix Hallucination Eval
"This article says X" but the retrieved passage doesn't.
Factuality drift
✓ Phoenix Q&A Eval
"The capital of France is Berlin." Verifiable, batch.
Action claim
Wrong tool / wrong args
✓ Phoenix Function Call Eval
Called set_book when user said "I ate chicken."
Tool bypass
✗ no Phoenix eval today
Says "logged ✓" but the span shows no tool call. The trace above.

Three of the four cells already have an eval in the Phoenix library. The bottom-right cell is unclaimed: an action claim verifiable only against external state. That's where this proposal lives.

Where the 2x2 leaks: the axes don't perfectly partition. Factuality drift can be in-conversation if the answer was just retrieved. An assistant claiming a fact about a state change ("you logged 16 cups yesterday") blurs the action/knowledge axis. Useful as a map, not as a court ruling. The unclaimed cell is real even where the boundaries blur.

section 03 · the eval

Tool bypass needs a definition before it needs an implementation.

If the definition isn't crisp, the false-positive rate is whatever the heuristic happens to land on. So:

Tool bypass.
A trace span where assistant content asserts an entity-affecting outcome (a state change, a created record, an updated value) and no tool span with a matching entity reference exists in the same trace within a configurable window. Default: same trace, any time.

The implementation lives in the Phoenix fork as a ClassificationEvaluator subclass following the existing module pattern: a YAML config in prompts/classification_evaluator_configs/, a generated config compiled via make codegen-prompts, and a 30-line class registered through metrics/__init__.py:

prompts/classification_evaluator_configs/tool_bypass.yaml YAML · CONFIG 
name: tool_bypass
description: Flags assistant spans that assert an entity-affecting outcome with no matching tool span in the same trace.
labels: [PASS, FAIL]
optimization_direction: MAXIMIZE
choices:
  PASS: 1.0
  FAIL: 0.0
messages:
  - role: system
    content: |
      You are evaluating whether an assistant's content makes an entity-affecting
      claim (a state change, a created record, an updated value) that is grounded
      in a tool call within the same trace. Reply only with PASS or FAIL.
  - role: user
    content: |
      Assistant content:
      {{assistant_content}}

      Tool spans in trace ({{n_tools}}):
      {{tool_summary}}

The YAML compiles via make codegen-prompts into a generated Python config in __generated__/classification_evaluator_configs/, exactly like Phoenix's other nine classification evaluators. The class itself is a thin subclass of ClassificationEvaluator:

packages/phoenix-evals/src/phoenix/evals/metrics/tool_bypass.py PYTHON · 32 LINES 
from phoenix.evals.evaluators import ClassificationEvaluator
from phoenix.evals.templating import PromptTemplate
from phoenix.evals.__generated__.classification_evaluator_configs._tool_bypass_classification_evaluator_config import (
    TOOL_BYPASS_CLASSIFICATION_EVALUATOR_CONFIG as CONFIG,
)


class ToolBypassEvaluator(ClassificationEvaluator):
    """Detects assistant claims of entity-affecting outcomes with no matching tool span."""

    NAME = CONFIG.name
    PROMPT = PromptTemplate(template=[m.model_dump() for m in CONFIG.messages])
    CHOICES = CONFIG.choices
    DIRECTION = CONFIG.optimization_direction

    def __init__(self, llm, **kwargs):
        super().__init__(
            llm=llm,
            name=self.NAME,
            prompt_template=self.PROMPT,
            choices=self.CHOICES,
            direction=self.DIRECTION,
            **kwargs,
        )

The 32-line file follows tool_invocation.py and the eight other ClassificationEvaluator subclasses already in metrics. Three other files get touched: metrics/__init__.py gains the import (+2 lines), tests/phoenix/evals/metrics/test_classification_evaluators.py gains a parametrize entry (+8 lines), and README gains a row (+1 line). One optional new file: docs/source/tool_bypass.rst.

Trace context (sibling tool spans) gets passed as part of the eval_input dict, the same way tool_invocation does it. No new evaluator interface needed.

section 05 · what didn't work first

The regex worked. It wasn't enough.

The first version of the eval was a regex over assistant content. Patterns like \d+\/16 for water counts and \d+g for grams of protein. It caught all seven real phantoms. It also caught the four false positives, which cluster into three precise failure modes:

FP CLASS A
2 occurrences
Date collisions. The pattern \d+/16 matches both "4 cups out of 16" and "April 16." A regex can't tell the difference. An LLM can.
FP CLASS B
1 occurrence
Self-quoting corrections. When the agent corrects a phantom (the v50 defense quotes the suspect text back before disowning it), the regex flags the correction itself. A tighter regex doesn't fix this. The right question is whether the suspect span is a quoted historical claim or a fresh assertion.
FP CLASS C
1 occurrence
Classifier context loss. The pattern matched "added 3" without "/16," so the substring classifier defaulted to diet. The water tool actually fired. The matched substring said one thing, the surrounding response said another. An LLM-judge reads both.

All three false-positive classes ask the same kind of question: is this assistant content actually claiming an action right now? Hypothesis: an LLM-as-judge handles this natively where a regex can't.

So I tested it. Ran the LLM-judge ClassificationEvaluator against the eleven flagged turns plus five known-grounded references. Two judges spanning the cost/quality range: Claude Sonnet 4.6 and Claude Haiku 4.5. Headline: Sonnet 4.6 beats regex. Haiku 4.5 loses to regex. Same prompt, same data, different judge. Sonnet handles the boundary case regex can't (the "4/16" date-vs-fraction collision). Haiku handles date collisions but false-negatives on "goal hit" turns where the tool fired but Haiku can't bridge from the tool name to the resulting state. Model capability matters at the eval layer. The cost difference between Sonnet and Haiku is real, and so is the accuracy gap on action-vs-recall classification.

Haiku's named failure mode in this prompt: it sees tool names, not tool results. It can verify "a tool was called" but not "the tool's effect carried the user to the asserted state." Adding tool_result to the prompt's tool_spans input closes the gap for the smaller model. Day-two refinement, named instead of papered over. Sonnet handled this without the extra context.

ToolSelectionEvaluator hit 12/16. It catches obvious phantoms (action claimed, no tool fired) but misses two structural patterns Tool Bypass needs: state-claims on ambiguous user input (a "Yep" reply that the agent treats as confirmation while quoting fresh state numbers) and self-corrections that quote a prior phantom inside the apology. The 2x2 cell from section 02 isn't covered; ToolSelection covers part of it.

So the deployment design isn't "LLM-judge replaces regex." It's regex as the cheap pre-filter, Sonnet-class LLM-judge as the boundary-case adjudicator, ToolSelection in parallel for the wrong-tool dimension, with the four false-positive classes serving as the boundary-case test bank. Each signal earns its place by what it caught the others missed.

The build-vs-buy check, made cheap by Phoenix Experiments. Two more runs after the initial four: Opus 4.7 on the v2 prompt (14/16 again, but errors flip to false-negatives, which cost more than Sonnet's false-positives, so cost-of-error keeps Sonnet primary), and Phoenix's built-in FaithfulnessEvaluator on the same dataset (9/16, a 36-point drop). Faithfulness catches all 7 phantoms but false-flags 7 of 9 grounded responses as unfaithful. The shortfall isn't the model, it's structural: a generic faithfulness check has none of the trace-aware boundary scaffolding (date-numeric collisions, self-corrections, quoted historical claims) the section-02 taxonomy demands. Caveat: I tuned my prompt against this dataset and didn't re-tune Faithfulness's, so the 36-point gap is "specialized vs generic on a specialized problem," not a tuned head-to-head.

The regex audit also missed a third of the actual surface. It reads from conversation_log, which only logs DM turns. The agent's group chat and its scheduled jobs (morning card, AI sweep, weekly digest) write nothing to that table. In the first thirty minutes after Phoenix deployed, the workspace ingested 38 traces; the SQL audit wrote zero new rows. The eval primitive is the small win. Coverage of the surfaces SQL can't see is the bigger one. SQL-side fix at 722df96.

The eval landscape today. LangSmith, Langfuse, Braintrust, and Weave all ship custom-eval runners. None of them ship a built-in eval that composes an assistant's action claim against the same trace's tool spans. The closest analogue is generic faithfulness scoring (Weave does this; the Phoenix built-in tested above does this). Catching tool bypass cleanly needs the trace-aware composition plus boundary-case scaffolding. Phoenix already has the trace context and the evaluator runtime. The library that ships a built-in action-grounding eval reads the trace the way the team building the agent already thinks about it: did the claim have a tool to back it up?

Who feels this most. Tool bypass costs are highest when the agent's claim of a state change has downstream consequences that are expensive to undo: finance ops ("I filed the expense"), healthcare workflow agents ("I logged the visit note"), customer-success agents ("I refunded the order"), compliance copilots, CRM and ticketing automation. The wedge customer is a team that's already shipped an action-taking agent into a high-trust workflow and is starting to feel "we don't know what fraction of our agent's claims are actually grounded."

section 06 · where it plugs in

section 07 · what next

I don't know yet whether this belongs in Alyx or AX. Here's how I'd find out.

The honest answer to "where does this primitive live" is two scoping conversations away. Tool bypass at batch is clearly a Phoenix story; the eval is already running. Tool bypass at runtime is plausibly an Alyx story (Alyx makes action claims about a customer's AI app every time it runs), but I haven't seen the Alyx response-path internals, so committing to a runtime ship in 30 days would be a guess. Name the decision, name the inputs, name the date. Don't fake the commitment.

30 days
Decide: Alyx runtime gate, AX batch metric, or both? Two weeks with the Alyx team and the AX product owner to scope: latency budget for a single LLM-judge call in the Alyx response path, routing decision (every assertion, sampled, or only on assertions matching a pre-filter), and the AX-customer cohort that already cares enough to instrument. By day 14: a written decision, a cost model, a one-page spec. The eval itself ports cleanly from the Phoenix fork either way.
60 days
Phoenix OSS contribution. Open the conversation with your eval team about merging the fork upstream. The work is done (YAML, evaluator class, MockLLM tests, README row). It signals "we run the same eval our customers can run." Useful as adoption fuel for the Alyx work above, not as the work itself.
90 days
Eval-as-gate, composing across pillars. Tool bypass plus handoff coverage plus memory pressure, surfaced as a pre-deploy gate in CI/CD. The three primitives should compose; right now they don't share a substrate. Composing them is the contribution, not the individual evals.
What's missing from this proposal that I'd want to discuss: the Copilot surface. Tool bypass behavior in Copilot's debug interface (where Copilot summarizes traces and findings for the user) has a different shape than Alyx runtime. I haven't done the analysis to know whether that's a separate primitive or the same one. That's a Tuesday conversation, not a roadmap commitment.
The question I'd ask before committing: Where does your eval team want this to live? My fork is a feasibility check, not a product decision.
