section 01 · what it does

Five users, twenty-seven tools, one daily card.

75 Hard is a 75-day habit challenge. Closed Telegram group of five friends plus Luke. Each person hits five daily tasks. Miss any one, the day is failed and the count restarts.

section 02 · how it's built

Anthropic, Telegram, SQLite, and now Phoenix.

One message in, one card out. Anthropic decides which tool to call. SQLite holds state. The scheduler fires nudges on each user's local clock. Phoenix wraps the LLM call so every span is queryable.

message handling pipeline · hover any step
01
Telegram message
DM or group · text, photo, or button tap
Two surfaces. DMs go to the user's per-user agent (in Bryan's voice). Group messages route to a shared chat agent for cards and milestones.
02
Handler
routes by chat type · loads user context
Routes to luke_chat (DMs) or the group handler. Loads conversation history and the active prompt. Short-circuits on filters before any LLM call (this is where Handler Bypass lives).
03
Anthropic SDK
claude-sonnet-4-5 · 27 tools available
Wrapped by openinference-instrumentation-anthropic. Every messages.create call becomes one root LLM span with the full tool catalog, input messages, output messages, and stop_reason as attributes.
04
Tool dispatch
tool_use blocks → Python functions
The model returns zero or more tool_use blocks per turn. Each one routes to a Python function (log_water, log_food, etc). When the model claims an action without a tool_use block, that's Tool Bypass.
05
SQLite mutation
tasks · daily_checkins · diet_log
Writes to data/75hard.db. Six tables: users, daily_cards, daily_checkins, diet_log, book_log, tasks. The conversation_log table is the legacy audit substrate Phoenix replaces.
06
Card re-render
image render · DM or group reply
Renders the user's daily card image (PIL). Posts to the original chat. Same surface for completion DMs, backfill confirmations, and the morning group card.
↑ phoenix wraps these two steps · one root span per turn
Scheduler runs alongside (not in this path): morning cards 7am ET, 9pm AI sweeps, 10pm local nudges, midnight PT lock.

section 03 · the instrumentation

From conversation log to traces, in one afternoon.

SQLite was fine for five users. The conversation_log table plus a regex audit script caught most of what mattered. The problem was scale and coverage: 100% of 175 logged rows were DMs, zero scheduled emissions, zero group chats. Phoenix saw 38 traces in the same window the audit script saw zero. The pipeline was scanning a third of the surface and didn't know it.

Phoenix earns its place when the surface gets too wide for a regex to keep up. For Luke at five users, that point came as soon as we wired in scheduled jobs. For an agent at fifty users, or one hundred, the gap between "I have a log" and "I can ask the log questions" is the whole product.

what one trace looks like · OpenInference Anthropic instrumentor
LLMmessages.createclaude-sonnet-4-5 · root span per turn
One root LLM span per Anthropic call. The Anthropic instrumentor is monkey-patched onto the SDK, so all five construction sites in Luke get instrumented automatically. No per-callsite wiring.
attrllm.input_messagesuser turn + 3 prior turns
User text plus conversation history loaded by the handler. Phoenix renders these as a structured chat view in the UI.
attrllm.tools[0..26]tool catalog at call time
Full JSON-schema for every tool the agent had access to on this call. Twenty-seven entries, one per registered tool. Lets you reason about "what could the agent have done" not just "what did it do."
attrllm.output_messagesassistant text + tool_use blocks
The model's response. May contain assistant text, zero or more tool_use blocks, or both. Tool calls live as attributes of this single LLM span. They are not separate child spans. That is by design in the OpenInference Anthropic instrumentor.
attrllm.stop_reasonend_turn · tool_use · max_tokens
Why the model stopped generating. end_turn means the assistant finished a normal turn. tool_use means it returned tool calls. max_tokens is the truncation case Luke's v50 retry catches.
Tool dispatch and SQLite mutation happen after the LLM span closes. They're separate code paths in Luke that don't currently emit their own spans. Custom SpanKind.TOOL emission was scoped out (see /proposal for why the flat structure already exposes everything the Tool Bypass eval needs).

Running the eval against the traces

  • Curated 16-case dataset (tool_bypass_validation_v1) from the audit ground truth: 11 flagged turns plus 5 grounded turns the regex correctly didn't flag
  • Regex baseline against the audit corpus of 169 historical DM turns: 11 flagged, 7 real phantoms, 4 false positives (full breakdown in /proposal)
  • LLM-judge variant landed as Phoenix Experiments on the 16-case set: Sonnet 4.6, Sonnet 4.6 with a v2 prompt pulled from the Prompt Registry, Opus 4.7
  • Final judge registered as a first-class Phoenix evaluator: tool-bypass-judge prompt with versioned history, Opus 4.7 binding, dataset linked
section 04 · the headline finding

Three judges held the same 14/16 ceiling on the same sixteen cases.

Sixteen cases pulled from the production audit (11 flagged, 5 grounded, hand-verified). Run against three LLM-judges plus Phoenix's off-the-shelf FaithfulnessEvaluator. All four landed as Phoenix Experiments against the same dataset.

Source: screenshots/11-three-experiment-metrics.png and screenshots/14-four-evaluator-metrics.png. Phoenix workspace, Datasets > tool_bypass_validation_v1 > compare.
  • The 14/16 ceiling holds across Sonnet, the v2 prompt, and Opus. Pushing the prompt into the Prompt Registry with explicit guidance for the boundary cases changed the explanations the judge wrote but didn't move the score.
  • The Tool-Bypass-specific judge beats off-the-shelf Faithfulness 14/16 to 9/16. Faithfulness flags the right phantoms but over-fires on grounded cases. The boundary cases this prompt was written for (quoted corrections, date collisions, prior-state recalls) are exactly what a generic judge misses.
  • Opus is roughly 25% faster than Sonnet on this task. 2.8s p50 versus 3.8s. Same prompt, same dataset.
  • Sixteen cases is a unit test. The dataset was sized to validate the boundary-case prompt against the production audit. The 14/16 number means "the prompt holds on the cases we already know about." A real benchmark would need an order of magnitude more cases and a different ground-truth process.
section 05 · what the audit found

Before instrumentation, a regex audit on Luke's log caught seven phantoms.

  • 30 days, 5 users, 169 turns → 11 flagged, 7 real phantoms + 4 false positives
  • All 7 predated Luke's v50 defense. Six days post-v50: zero
  • 6 of 7 went undetected in the moment. Users acted on phantom state and recovered hours later via verification questions
  • First 30 min of Phoenix: 38 traces. Same window in conversation_log: 0. Audit was scanning a third of the surface

Two failure shapes, one visible to Phoenix

where each bypass lives in the pipeline · hover any step

Tool Bypass

covered · 14 of 16 by the LLM judge
01User message arrives. Routed to luke_chat.
Standard DM. Nothing special yet.
02Handler loads context, calls the model.
Conversation history, user state, and the tool catalog get passed to messages.create. Phoenix begins its span here.
03Anthropic responds with assistant text only.
No tool_use blocks in the response. The model says something like "I've logged your water" but doesn't actually request log_water_dm.
04No tool dispatch. No SQLite write.
Without a tool_use block, there's nothing for the dispatcher to call. The state-change claim is a phantom.
05Reply sent to user. Belief diverges from state.
User reads "logged your water" and trusts it. The water count never updated. Catches up hours later when the user verifies via the card or status query.
Phoenix sees the LLM span. The Tool Bypass eval can read the assistant text and the empty tool-use list and flag the gap.

Handler Bypass

invisible · the proposal's next primitive
01User message arrives. Routed to luke_chat.
Same path so far. Real example: "Luke, show me a picture of what you look like."
02Handler short-circuits before reaching Anthropic.
A filter, a guardrail, an early return, a malformed-content check, a non-text payload Luke can't handle. Code path exits before messages.create.
03No LLM call. Phoenix has nothing to wrap.
The instrumentor only fires when the SDK is called. Skip the SDK, skip the trace.
04No reply. No tool dispatch.
User sent a message. Luke didn't act, didn't acknowledge, didn't say why.
05User waits on a silent agent.
From the user's perspective, Luke ignored them. No error message, no graceful "I can't do that," nothing.
Phoenix sees nothing. The next primitive needs visibility into the handler itself, not just the LLM call.
Tool Bypass is the one this take-home addresses. Handler Bypass is the natural sibling. Both belong to the same agent-failure taxonomy; one is currently in scope, the other is the most useful next eval Phoenix could ship.
