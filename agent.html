section 01 · what it does

Six users, twenty-seven tools, one daily card.

Luke runs the 75 Hard challenge for a closed Telegram group. Five daily tasks per user (two workouts, water, diet, reading, photo). One agent per user, holding a long-running DM conversation in Bryan's voice. One shared group chat where the agent posts daily cards and milestones.

It's a real production agent. Six users complete the challenge in real time. The agent handles backfills, undo, food vision, transformation timelapses, and the daily rhythm of nudges. It also lies sometimes. That's the trace at the heart of the proposal.

section 02 · how it's built

Anthropic, Telegram, SQLite. And now Phoenix.

The stack is small enough to fit in a single diagram. Telegram polls for messages. The bot dispatches them to handlers. The DM handler hands off to the LLM with the user's history loaded from SQLite. The LLM picks tools. Tools mutate state. Cards re-render. A scheduler fires nudges at user-specific timezones. A daily AI sweep posts a "spicy moment" when the group chat warrants one.

As of today, every Anthropic call is wrapped by openinference-instrumentation-anthropic. Tool-call dispatch emits custom SpanKind.TOOL spans with tool.name and tool.parameters attributes. Traces ship to a Phoenix Cloud workspace at app.phoenix.arize.com/s/bsplaza.

section 03 · the instrumentation

From conversation log to traces, in one afternoon.

Until today, Luke's introspection surface was a SQLite table called conversation_log. Useful but flat: rows of role, content, tool calls, timestamps. The audit script described in the proposal walked that table with regex.

This afternoon I wired in arize-phoenix-otel and openinference-instrumentation-anthropic, then deployed to production. Every Anthropic messages.create call emits an LLM span with the full tool catalog, the model's tool_use blocks, and stop_reason as attributes. The same shape the audit script was reading from SQL is flowing as OTLP into a Phoenix Cloud workspace, ready for the Tool Bypass eval to consume.

Deploy time: 2026-05-01 22:50 UTC. Image deployment-01KQJVQY3D7F4TKD6J8FGGYFK8. As I write this, traces from real user conversations are landing in the workspace.

Then I ran the evaluator against those traces.

The eval primitive in the proposal isn't just a code artifact. After Phoenix deployed and prod traffic flowed in, I ran a regex-based Tool Bypass classifier against all 57 production spans in the project, then pushed the results back as Phoenix annotations via client.spans.log_span_annotations_dataframe. Phoenix lifted the result into a project-level metric in the same row as Total Cost and Latency.

Result: tool_bypass μ 1.00. All 57 turns grounded their action assertions in tool calls. Zero phantoms.

The same regex variant, run earlier against the 169 historical pre-v50 DM turns in conversation_log, flagged 11 suspect turns: 7 real phantoms and 4 instructive false positives (full table in /proposal section 04). The eval catches the failure when the failure exists; passes cleanly when the layered defense holds.

I also ran the full LLM-judge ClassificationEvaluator from the Phoenix fork against the same eleven flagged turns plus five known-grounded references, with two judges to test the cost/quality spectrum: Claude Sonnet 4.6 and Claude Haiku 4.5. Result: Sonnet 14/16 (88%), beating regex; Haiku 11/16 (69%), losing to regex. Same prompt, same data. Phoenix's existing ToolSelectionEvaluator as a baseline: 12/16 (75%). Sonnet correctly handled the boundary case regex couldn't (the "4/16" calendar-date collision); Haiku managed that one too but false-negatived on "goal hit" turns where the tool fired but it couldn't bridge to the resulting state from the tool name alone. Model capability matters at the eval layer. Detailed comparison table in /proposal section 05.

05-tool-bypass-eval-running.png Phoenix project header with tool_bypass μ 1.00 alongside Total Cost and Latency. Per-row annotations populate the Annotations column.

Then I made the validation set durable as a Phoenix Dataset, and the run reproducible as a Phoenix Experiment.

Comparing four evaluators in a notebook is a one-shot artifact. The next person who wants to test a different prompt, a different model, or a different temperature has to set up the harness from scratch. Phoenix has primitives for exactly this gap: Datasets (a fixed set of examples with ground truth) and Experiments (a saved run of a task + evaluators against a Dataset, comparable side-by-side in the UI).

I uploaded the 16-turn validation set as a Phoenix Dataset called tool_bypass_validation_v1 via client.datasets.create_dataset. The 11 regex-flagged turns (7 true phantoms, 4 false-positive boundary cases across the three named classes) plus 5 known-grounded reference turns. Each row carries the input, the assistant output, the tool spans observed in the trace, the available tool surface, the ground-truth label I hand-graded, and metadata for the false-positive class.

Then I ran the LLM-judge as a Phoenix Experiment via client.experiments.run_experiment: the task is the ClassificationEvaluator with Sonnet 4.6 returning {label, score, explanation}; the evaluator is a matches_ground_truth function comparing the judge's label to the dataset's ground truth. Result: 14/16 match (87.5%), the same two mismatches the notebook run produced. The two failures were on the boundary cases all four evaluators struggled with: #41 (jailbreak attempt with confused tool sequencing) and #194 (self-correcting follow-up that quotes a prior phantom).

The Experiment is saved in Phoenix. The next iteration of the eval, whatever it is, points at the same Dataset, runs as a new Experiment, and shows up next to tool_bypass_sonnet_4_6 in the UI's compare view. That's the platform value: regression testing for evals, treated as a first-class workflow rather than a notebook habit.

06-phoenix-experiment.png Phoenix Experiments grid for tool_bypass_validation_v1: the tool_bypass_sonnet_4_6 run is selected, with the matching-ground-truth evaluator scored 1.00 on the visible row, AVG 3.8s judge latency, and the dataset's input/reference output columns alongside the experiment's output.
section 04 · what the audit found

Before instrumentation, a regex audit on Luke's prod log caught seven real bugs.

Over thirty days of production traffic across five active users, the audit script flagged eleven suspect turns. Hand-graded, seven were real phantom-action bugs. Four were false positives that clustered into three named failure modes: date-numeric collisions, self-quoting corrections, and classifier context loss.

Those seven phantoms predated Luke's v50 layered defense. After v50 (Apr 24 onward), the audit found zero new phantoms in six days of production traffic.

The full audit table, with the eleven flagged rows, ground truth, and false-positive class labels, lives in the proposal section. The script that produced it is in Luke's repo at scripts/audit_conversations.py. The Tool Bypass evaluator in the Phoenix fork is a productized version of the same logic, ready to run on traces flowing into the Phoenix workspace right now.

What the audit didn't see, and only Phoenix surfaced

Side-by-side with Phoenix, the regex audit revealed something it could not have flagged on its own. The audit reads from Luke's conversation_log SQLite table. Pulled the live prod DB this evening: 175 historical rows across two weeks, 100% with source='dm'. Zero rows tagged as group chat. Zero from scheduled jobs.

Luke's primary social surface is a group chat where five users talk to Luke and to each other. None of that traffic was being written to conversation_log. Neither were the morning card, the AI sweep, or the weekly digest, all of which call the LLM on a timer.

In the first thirty minutes after Phoenix deployed, the workspace ingested 38 production traces. Over the same window, conversation_log wrote zero new rows. The audit was scanning a third of the surface, confidently. Phoenix was scanning the whole thing the moment it turned on.

The eval primitive in the proposal is the small fix. The audit pipeline replacement is the bigger one. The same regex, the same evaluator, now covering the surfaces the SQL pipeline structurally couldn't see.

And the SQL pipeline itself just learned to see them. Same evening, the agent's main branch picked up commit 722df96: a log_scheduled_emission() helper writing source='scheduled' rows from every job + admin path, an audit_conversations.py summary header that prints by source: dm=N, scheduled=N, group=N, and 9 new tests asserting SELECT DISTINCT source returns the full set. 107/107 tests pass on main. Forward-only. The next deploy starts logging scheduled emissions; the next audit run shows the new shape. Phoenix and the SQL audit are now genuinely dual coverage, not "Phoenix sees, SQL guesses."

What the phantoms cost the users

I pulled the next three messages each user sent after every one of the seven historical phantoms. I was looking for the moment users noticed something was wrong. They almost never did.

Six of seven phantoms went undetected at the moment they happened. Users acted on the phantom state. Bryan logged more food after a "goal hit" celebration; Cam logged a workout after a phantom water confirm. The bug's cost isn't measured at the failure turn. It's measured in what users did downstream while trusting confirmed-but-fake state.

The recovery signal, when it came, was a verification question: "What have I eaten" (Bryan, 18 hours later) and "How many cups of water I at" (Kat, the next day). Those messages are themselves a measurable behavior, an Eval primitive in their own right: frequency of unprompted state-verification queries as a proxy for trust loss. A team like Arize ships the eval and the metric together. The eval catches the bug; the metric confirms users are acting on stale or wrong state.

And what Phoenix still doesn't see, even now

This evening, while writing this submission, I sent the agent a message in the group chat: "Luke, show me a picture of what you look like." No response. Total silence.

I checked Phoenix immediately. No new trace in the project for that turn. Which means the agent's handler short-circuited before reaching Anthropic. Probably the mention-detection or trigger-phrase routing logic decided this wasn't a request that needed the LLM. The current Phoenix instrumentation can't see that decision. It only sees Anthropic API calls; it has no spans at the handler entry point or at the Telegram-send egress.

Tool Bypass catches "called LLM, claimed action, didn't fire tool." This silent failure is one layer up: "received message, decided to skip LLM, never told the user why." Same shape, different layer. The natural extension of Tool Bypass into a Handler Bypass primitive is the day-two work this take-home is implicitly proposing. The instrumentation gap is real, observable, and the agent already has the failing example.
