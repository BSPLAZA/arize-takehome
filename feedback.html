Field notes from instrumenting.

I started with Phoenix, switched to AX, drove both end-to-end. What pulled me over, what worked, what broke, and what I'd change.

Bryan Plaza
2026-05-04
Reading time: 5 minutes
section 01 · why I switched

Phoenix is open source, private, free with no limits. I switched to AX anyway.

I picked Phoenix first for the right reasons. It's open source, private, and the free Cloud tier has no real ceiling for an agent at my scale. I shipped Luke into a Phoenix Cloud workspace, ran traces, ran experiments, registered an evaluator. Then I instrumented the same agent into AX and did the same thing.

I stayed in AX. The reasons are easier to read in my own words.

Bryan, mid-session, on the switch I started with Phoenix because it was open source, private, and free with no limits. The interface was really lacking. The features I wanted matter even if I didn't want to scale. The AX UI is friendlier, more playful, more intuitive. With gRPC I didn't have to deal with the finickiness of HTTP requests and getting them well-formatted, or asking AI to fix them.

Setup, both platforms, side by side

Phoenix Cloud

HTTP · workspace-prefixed
Endpoint trial-and-error. Bare domain returns 401. Workspace-only URL returns 200 and silently drops traces. The working shape is app.phoenix.arize.com/s/<workspace>/v1/traces.
Five ways to pass the API key. Client(api_key=), PHOENIX_API_KEY, header dict, Bearer header, PHOENIX_CLIENT_HEADERS. None flagged as recommended.
HTTP transport details surface in the SDK. When the URL is wrong, the failure mode depends on which wrong URL you tried. Two have to be diagnosed separately.
~10 minutes of trial and error to land traces.

Arize AX

gRPC · single header
Single endpoint. otlp.arize.com. No workspace path to remember.
Three required params. space_id, api_key, project_name. Worked first try.
Transport hidden from the developer. The package picks gRPC. Auth is one header. Wire-format details stay below the API.
Traces visible in < 60 seconds.

The transport choice is the smallest example of the larger pattern. Phoenix exposes the SDK as a thin wrapper over its REST + GraphQL surface. AX exposes higher-level primitives. For a builder shipping in a hurry, AX gets out of your way faster. For a builder customizing edge cases, Phoenix gives you more access. I needed the first.

section 02 · phoenix friction

What "the interface is really lacking" means concretely.

Five frictions from the Phoenix session. The 8-item raw list is at the bottom for anyone curious. These five are the ones that drove the switch.

01
The trace-to-eval handoff is invisible from the UI. The trace-detail annotations panel reads "No annotation configurations for this project." No CTA. No "run a built-in eval" button. The only path is a Python SDK script. Once you've created configs via REST, they still don't appear in this panel because the workspace-to-project binding isn't surfaced. Wire the empty panel to one-click: pick eval, pick model, run on this trace. The first eval should be a button, not a notebook.
02
Evaluator registration is a UI-only flow. No public REST or GraphQL mutation accepts prompt plus model plus dataset and lands an evaluator on /evaluators. I had to break out of the agent loop and click through the modal manually. Surface POST /v1/evaluators. Without it, AI-driven Phoenix workflows hit a wall exactly where the agent should keep going. Alyx will hit the same wall on AX customers' behalf.
03
The simplest evaluator primitives are buried. Found MatchesRegex after writing a custom regex classifier. Would have replaced 30 lines. Docs lead with the LLM judges. The lightweight primitives aren't on the same page. "Before you write your own evaluator" page listing MatchesRegex, ExactMatch, PrecisionRecallFScore. AX surfaces these as first-class registered entities; Phoenix has them but doesn't elevate them.
04
The filter bar expects raw query syntax. span_kind == 'LLM' as raw input. No query builder, no column-header click-to-filter, no autocomplete. Power-user friendly, beginner-hostile. Column-header dropdowns for kind, status, latency bucket. Keep the raw query for power users.
05
No agent assistant. No Monitors. No Dashboards. No Playgrounds. No Labeling Queues. The interface gap isn't one missing feature. It's a category of work the platform doesn't currently support without a notebook in the loop. AX has all of them. Each is its own roadmap conversation. Bundling matters: the absence of any one is fine; the absence of all of them is what defines the gap.
+ three more, smaller-impact
06
DataFrame columns aren't list-flattened the way the rest of the schema is. attributes.llm.output_messages returns a list of dicts. The flat attributes.llm.output_messages.0.message.content form isn't there. Extractor returned empty strings until I introspected. Flatten consistently, or surface the schema in the SDK docstring.
07
URLs use base64 GraphQL IDs. UHJvamVjdDox instead of luke-75-hard. Cosmetic but unfriendly to share. Some browser-automation tools refuse to navigate them. Add a slug alongside the base64. Default the URL bar to the slug.
08
Workspace endpoint trap. https://app.phoenix.arize.com returns 401. https://app.phoenix.arize.com/s/<workspace> returns 200 and silently drops spans. The working shape is /s/<workspace>/v1/traces. The Cloud onboarding panel should show this exact URL. Right now it shows the workspace-only form.
section 03 · what broke on ax

Honest report from inside the workspace.

Six items, in order of impact. Headline first: Alyx is a router, not an agent.

01 · Alyx is a router, not an agent.

Six prompts in a live Alyx session. Six page-handoffs in response. Alyx interprets the intent, identifies the right page, opens it, disengages. "Alyx will help you" inside the page is a UI handoff, not a continuation. Once you click in, Alyx is gone and you're filling out the form yourself.

What I asked AlyxWhat Alyx returned
"What failure modes do you see in this project?" Asked which project. After I picked luke-75-hard: "I've opened your luke-75-hard project where Alyx will continue helping you analyze failure modes." Stopped.
"What failure modes do you see in this project?" (repeated, project context already set) "I've opened your project page where Alyx will help you identify and analyze failure modes." Listed what the page can do. Did nothing itself.
"Look at the luke-75-hard project. What failure modes do you see?" "I can see your project, but I don't currently have tools to directly query and analyze trace data." Routed to the project page.
"Why did experiment tool_bypass_judge_sonnet_phoenix_parity only hit 9 of 16?" "I've opened the tool_bypass_validation_v1 dataset page where Alyx will help you investigate." No analysis.
"Can you build me an evaluator that catches tool_bypass?" "I've opened the evaluator creation page where Alyx will help you build a tool bypass detection evaluator." The actual creation happens in the form I now have to fill out.
"Set up a monitor that fires when tool_bypass score drops" "I've opened the Monitors page where Alyx will help you create an alert." Same shape.

The cost isn't one disappointing session. It's faith. After three "I've opened the page" replies, users stop asking Alyx anything substantive. By the time the agent improves, the muscle memory is "skip Alyx, just navigate."

What I'd change. Alyx should continue agentic flows in-context. When a user asks "build me an evaluator," Alyx should call evaluators.create_template_evaluator on the user's behalf, surface the result, ask follow-ups. If Alyx can't act, warm handoff with context: pre-fill the form with a draft prompt, pre-set the threshold from the conversation, pre-filter the traces panel. Right now the routing strips context on the way through.

02 · Labeling Queue: clicking Next discards your label.

I was annotating a labeling queue (16 disputed Tool Bypass cases). Each card has radio options for phantom or grounded, then two buttons: Save and Next. The radio fills blue when you click it, which feels like state. It's not. Selecting a radio doesn't persist. Save persists but doesn't advance. Next advances but doesn't persist.

I went back to verify a previous row and it was empty. To actually annotate every row, the reviewer has to click two buttons in sequence: Save, then Next. Sixteen rows is thirty-two clicks for what should be sixteen. No warning. No "unsaved changes" toast. The bug is silent.

What I'd change. Auto-save on selection (cleanest), or auto-save on Next (more conservative), or combine into one "Save & Next" primary action. Anything but silent data-loss on the human-review surface, which is the surface where data-loss is most expensive.

03 · Two evaluator surfaces, hand-rolled bridge.

AX has registered evaluators (client.evaluators.create_template_evaluator()) and inline experiment evaluators (LLMEvaluator subclass or callable). They look like the same thing. They are not connected. You cannot pass a registered evaluator's ID to experiments.run(). The registered evaluator is for online tasks and the UI; the inline one is for offline experiments.

The bridge is the prompt registry. Push the prompt content via prompts.create(), pull by name in your experiment script, subclass LLMEvaluator, fill the f-string template, call the LLM yourself. Tested. Worked. 13 of 16 with the registered prompt vs 9 of 16 with an inline hand-written one, same model, same data. The registered prompt was shorter and outperformed.

What I'd change. Add LLMEvaluator.from_registered_prompt("name") that returns a fully-configured evaluator. The opinionated common case isn't paved.

04 · Custom code evaluators are paywalled.

create_code_evaluator() takes two flavors. ManagedCodeConfig references one of the pre-built evaluators (regex, json-parseable, contains-any-keyword, contains-all-keywords). Works on Free. CustomCodeConfig uploads a Python function for AX to execute server-side. Returns 400 on Free: "Custom code evals are not available for your account."

The Free plan marketing list says "Evals (Offline & Online)" without qualifier. The actual gate is "managed evals only; bring-your-own-Python is paid." For the Tool Bypass scenario, that's the difference between shipping the eval as a registered code path or shipping it as a template (LLM-as-judge).

What I'd change. One line in the Free-plan marketing copy. "Evals (managed primitives + LLM-as-judge templates; custom Python on paid)." Honest disclosure helps developers pick the right tier.

05 · API maturity is uneven.

The SDK methods my workflow needed (experiments.list_runs, evaluators.create_code_evaluator, tasks.list_runs) all exist, all are decorated with @prerelease_endpoint(stage=ReleaseStage.ALPHA) or BETA, and every call to them prints a stderr warning. Writing a CI/CD pipeline against these APIs means accepting that response shapes can change. The opportunity is making the BETA-graduation path more visible to users, with a stability guarantee or a sunset notice when one moves.

06 · Indexing latency.

Trace ingestion has a noticeable indexing delay before spans become queryable via the SDK. Phoenix Cloud is near-realtime on the same workload. For someone debugging "did my span land," that gap is felt fast.

section 04 · what I'd push on

Three things, in order of conviction.

  • Make Alyx agentic, or warm-handoff with context. Today it's a slower way to navigate. Two acceptable fixes: tool-call into the SDK on the user's behalf, or pre-fill the page it routes to with the conversation context. Both fixes ship the value the agent assistant promises.
  • Fix the Labeling Queue Save/Next bug. Auto-save on Next, or combine the buttons. Silent data loss on the human-review surface is the most expensive bug shape this product can have. The reviewer is the customer's most expensive person; their work has to capture.
  • Pave the registered-evaluator-to-experiment bridge. Add LLMEvaluator.from_registered_prompt() so the natural mental model ("register once, use everywhere") actually holds. The prompt registry is already the bridge; it just needs a one-liner helper.
