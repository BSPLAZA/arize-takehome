section 01 · phoenix

Onboarding to first trace, in one afternoon.

Signed in at app.phoenix.arize.com via GitHub OAuth, generated an API key from General → API Keys, set PHOENIX_API_KEY and PHOENIX_COLLECTOR_ENDPOINT in Luke's .env. Quickstart followed almost verbatim: two pip installs, four lines of register() and AnthropicInstrumentor().instrument() in Luke's startup, deploy. The bot's messages.create calls started landing as LLM spans inside thirty seconds. I then pulled the live span set with client.spans.get_spans_dataframe(), ran a regex Tool Bypass classifier in Python, and pushed the verdicts back as annotations with client.spans.log_span_annotations_dataframe(). Phoenix lifted the result into a project-header metric next to Total Cost and Latency. That feedback loop, when it works, is the strongest thing about the platform.

The friction I hit between sign-in and seeing my eval go green:

What I hit, in order

01
Client(base_url=...) needs the workspace path or every call returns 401. The Cloud quickstart shows https://app.phoenix.arize.com as the endpoint. Passing that to Client returned 401 on the first get_spans_dataframe. Switching to https://app.phoenix.arize.com/s/bsplaza fixed it. Three attempts to land on the right shape. Accept the bare domain and infer the workspace from the API key (the JWT carries it), or surface a clear error: "this key is bound to workspace 'bsplaza'; set base_url to https://app.phoenix.arize.com/s/bsplaza." Right now the failure is a generic HTTPStatusError: 401.
02
API key plumbing has five working paths and no canonical one. Client(api_key=...) works. os.environ['PHOENIX_API_KEY'] works. headers={'api_key': key} works. headers={'Authorization': f'Bearer {key}'} works. PHOENIX_CLIENT_HEADERS=api_key=<key> for the OTLP exporter works. All valid, none flagged as recommended. Two minutes deciding which to use. Pick a recommended path and put it at the top of the SDK quickstart. The other four can be alternates.
03
DataFrame columns aren't list-flattened the way you'd expect. get_spans_dataframe() returns a column attributes.llm.output_messages whose values are Python lists of dicts, not flattened to attributes.llm.output_messages.0.message.content the way attribute paths usually are. My first classifier returned empty strings for every row because the dotted path didn't exist. Three minutes to introspect the columns and rewrite the extractor. Flatten consistently, or surface the schema in the get_spans_dataframe docstring. Current behavior is correct but invisible until you hit it.
04
The simplest evaluator primitive is buried. I wrote a custom regex classifier. After it worked, I ran dir(phoenix.evals.metrics) out of curiosity and found MatchesRegex. It would have replaced thirty lines of my code. Docs lead with the LLM-judge evaluators (Hallucination, Faithfulness, Refusal); the lighter-weight building blocks aren't on the same page. Add a "before you write your own evaluator" page. List the primitives (MatchesRegex, ExactMatch, PrecisionRecallFScore) with one-line use cases. Saves the next person thirty lines.
05
The trace-to-eval handoff is invisible from the UI. The trace-detail annotations sidebar reads "No annotation configurations for this project." No setup CTA. No "run a built-in eval on this trace" prompt. The only way I got annotations into the UI was via the Python SDK in a separate script. From inside the UI, you can't. Wire the empty annotations panel to a one-click flow: pick a built-in eval, pick a model, run on the current trace. The first eval should be a button, not a Python script. The SDK path stays for batch and CI.
06
The filter bar expects you to already know the query syntax. The trace list filter takes span_kind == 'LLM' as raw syntax. No query builder, no column-header click-to-filter, no autocomplete chips. New users guess or read a doc. Column-header dropdowns for the common filters (kind, status, latency bucket). Keep the raw-query input for power users. The first session shouldn't require a doc.
07
URLs use base64 IDs instead of human-readable slugs. Project and trace URLs are base64-encoded GraphQL global IDs (UHJvamVjdDox). Cosmetic, but it makes URLs unfriendly to share in chat or screenshots, and some browser-automation tools refuse to navigate them as a defense. Add a slug or short-id alongside the base64. Default to the slug in the URL bar. Keep base64 as the canonical id for the API.
08
Annotation Configs are workspace-wide via REST, but the trace-detail "Edit Annotations" panel needs a per-project binding I couldn't reach via the public API. I created tool_bypass as a categorical annotation config via POST /v1/annotation_configs (200 OK). Pushed annotations onto recent spans via log_span_annotations_dataframe (200 OK). The trace-detail Annotation Summary shows them rolled up. But the Edit Annotations panel right below reads "No annotation configurations for this project." The schema and the data flow work; the click-to-annotate UI doesn't pick them up. POST and PUT to /v1/projects/{id}/annotation_configs both return 405. So a reviewer in the trace UI still can't structurally label a span without a Python notebook. Auto-bind workspace configs to all projects, expose the binding endpoint over REST, or surface a "use in this project" toggle on the workspace settings page. The fix is small. The impact is real: structured annotations stop being a Python-only workflow and become reachable for any reviewer in the trace detail UI.

What worked well

The "Waiting for traces to arrive" empty state is the strongest part of onboarding. It puts the integration picker and code snippet on screen immediately. Most products fail this moment with a marketing splash or a docs link; Phoenix shows the next action.

The Stream toggle (live trace feed) is well-positioned for the developer's first "did it work?" moment. The span-kind chips (color-coded agent, chain, tool) make the table fast to scan. Python and TypeScript get equal billing from day one.

section 02 · ax

An outside-in read on the surfaces I'd actually work on.

This half is from documentation, product pages, and the Alyx demo video. I did not run a paid trial. I'm not pretending the friction below is from inside the product. It's the read a PM does before a first product review.

What I'd want before joining: a seat in AX. The proposal leads with Alyx for a reason. If I joined the team, the first thirty days would be spent inside that surface, not outside it. The notes below are what I'd bring to the first meeting, not the second.

What I observed

01
Alyx's empty state: what does it do for me on day one? From the demo and docs, Alyx is positioned as an AI agent that helps users debug their AI app. The first-load empty state isn't clear from the public surface. What does Alyx know about my app before I've explained anything? Does it auto-discover from my traces, or wait for me to type? A PM hire question, not a feedback note: what's the time-to-value loop for Alyx, and is empty-state agency the right place to invest? Tool Bypass at runtime in Alyx (the proposal's 30-day target) shifts that loop from "what do I ask Alyx" to "Alyx tells me before I ask."
02
Copilot's role versus Alyx's role isn't obvious from the marketing surface. The product pages mention both. The line between them (Copilot as a debug companion in the product, Alyx as an agent on the customer's app) is implied but not stated. A positioning question I'd want to debate internally. The shortest version of the answer ("Copilot helps you understand traces; Alyx helps your app evaluate itself") would shorten the customer onboarding conversation.
03
The Bedrock AgentCore partnership is significant and barely visible. AX shipped a partnership with AWS Bedrock AgentCore. From the product surface, it reads as a supported integration. From the customer-decision surface, it's a major signal: Bedrock customers don't have to choose between the AWS-blessed eval path and AX. If this is the wedge for the AWS-anchored enterprise segment, treat it like the wedge. Current copy reads as "another integration." Promote it.
04
The customer case studies are technical, not commercial. The case-study pages described implementation: which models, which evals, which features. They didn't describe commercial outcomes: adoption inside the customer's company, time-to-value, second-quarter renewal signals. Add a top-line outcome metric to each case study (time-to-first-eval, percent of teams adopted, weeks-to-second-product-line). Technical depth is good. Commercial framing is what converts the next prospect.

section 03 · what I'd push on

Three things, in order of conviction.

One. The trace-to-eval handoff in Phoenix should be a one-click flow from any trace, not a docs page. If the user can't get from "this trace looks suspicious" to "this eval said FAIL with this evidence" in two clicks, the eval library doesn't earn its place above the trace. Highest first-month impact for new Phoenix users.

Two. Alyx needs a runtime grounding gate. The proposal makes this case. Tool Bypass at runtime, applied to Alyx's own action assertions, turns "trust me, the AI agent helped" into "here's why the AI agent's claim is grounded." Same primitive that Phoenix ships in OSS, lifted into runtime in AX. That composition is what justifies the AX premium tier.

Three. Customer case studies need commercial framing, not technical framing. Time-to-first-eval, percent-of-teams-adopted, weeks-to-second-product-line. Technical depth is there; commercial signal isn't. Future prospects should read a case study and learn the buyer's outcome, not the buyer's stack.
