how I gathered this

Two halves, two methods, both honest about which is which.

section 01 · phoenix

Onboarding to first trace, in one afternoon.

Signed in at app.phoenix.arize.com via GitHub OAuth, generated an API key from General → API Keys, set PHOENIX_API_KEY + PHOENIX_COLLECTOR_ENDPOINT in Luke's .env. Quickstart followed almost verbatim: two pip installs, four lines of register() + AnthropicInstrumentor().instrument() in Luke's startup path, deploy. The bot's messages.create calls started landing as LLM spans inside thirty seconds. After the deploy I pulled the prod span set with client.spans.get_spans_dataframe(), ran a regex Tool Bypass classifier in Python, and pushed the verdicts back as annotations with client.spans.log_span_annotations_dataframe(). Phoenix lifted the result into a project-header metric next to Total Cost and Latency. That feedback loop, when it works, is the strongest thing about the platform.

What follows is the friction I hit between sign-in and seeing my eval go green.

What I hit, in order

01
Client(base_url=...) needs the workspace path or every call returns 401. The Cloud quickstart shows https://app.phoenix.arize.com as the endpoint. Passing that to Client got me a 401 on the first get_spans_dataframe. Switching to https://app.phoenix.arize.com/s/bsplaza fixed it. Three attempts to land on the right shape. Either accept the bare domain and infer the workspace from the API key (the JWT carries it), or surface a clear error: "this key is bound to workspace 'bsplaza'; set base_url to https://app.phoenix.arize.com/s/bsplaza." Right now the failure is a generic HTTPStatusError: 401.
02
API key plumbing has five working paths and no canonical one. Client(api_key=...) works. os.environ['PHOENIX_API_KEY'] works. headers={'api_key': key} works. headers={'Authorization': f'Bearer {key}'} works. PHOENIX_CLIENT_HEADERS=api_key=<key> for the OTLP exporter works. All valid, none documented as the recommended one. Two minutes spent finding which to use. Pick a recommended path and put it at the top of the SDK quickstart. The other four can be alternates further down.
03
DataFrame columns aren't list-flattened the way you'd expect. get_spans_dataframe() returns a column named attributes.llm.output_messages whose values are Python lists of dicts, not flattened to attributes.llm.output_messages.0.message.content the way attribute paths usually are. My first classifier returned empty strings for every row because the dotted path didn't exist. Cost: ~3 minutes to introspect the actual columns and rewrite the extractor. Either flatten consistently, or surface the schema in the docstring of get_spans_dataframe. The current behavior is correct but invisible until you hit it.
04
The simplest evaluator primitive is buried. I built a custom regex classifier from scratch. After it was working, I ran dir(phoenix.evals.metrics) out of curiosity and discovered MatchesRegex. It would have replaced thirty lines of my custom code. The docs lead with the LLM-judge evaluators (Hallucination, Faithfulness, Refusal); the lighter-weight building blocks aren't on the same surface. Add a "before you write your own evaluator" page. List the primitives (MatchesRegex, ExactMatch, PrecisionRecallFScore) with one-line use cases. Save the next person thirty lines.
05
The trace-to-eval handoff is invisible from the UI. The trace detail panel's annotations sidebar reads "No annotation configurations for this project." No setup CTA. No "run a built-in eval on this trace" prompt. The way I actually got annotations into the UI was via the Python SDK in a separate script. From inside the UI, you can't. Wire the empty annotations panel to a one-click flow: pick a built-in eval (Hallucination, Function Call, Tool Bypass), pick a model, run on the current trace. The first eval should be a UI button, not a Python script. The SDK path stays for batch and CI.
06
The filter bar expects you to already know the query syntax. The trace list filter input takes span_kind == 'LLM' as raw filter syntax. No query builder, no column-header click-to-filter, no autocomplete chip flow. New users either guess or read a doc. Column-header dropdowns for the common filters (kind, status, latency bucket). Keep the raw-query input for power users. The first session shouldn't require a doc.
07
URLs use base64 IDs instead of human-readable slugs. Project and trace URLs are base64-encoded GraphQL global IDs (UHJvamVjdDox). Cosmetic, but it makes URLs unshareable in chat or in screenshots, and the security layer in some browser-automation tools refuses to navigate them as a defense. Add a slug or short-id alongside the base64. Default to the slug in the URL bar. Keep the base64 as the canonical id for the API.

What worked well

The "Waiting for traces to arrive" empty state is the strongest part of the onboarding. It does not leave the user stranded. It puts the integration picker and code snippet immediately on screen. Most products fail this moment by showing a marketing splash or a docs link; Phoenix shows the next action.

The Stream toggle (live trace feed) is well-positioned for the developer's first "did it work?" moment. The span-kind chips (color-coded agent, chain, tool) make scan-reading the table fast. Python and TypeScript get equal billing from day one. No language hierarchy.

section 02 · ax

An outside-in read on the surfaces I'd actually work on.

This half is from documentation, product pages, and the Alyx demo video. I did not run a paid trial. I'm not pretending the friction below is from inside the product. I'm pretending it's from a PM doing the kind of read I'd do before a first product review.

What I'd want before joining: a real seat in AX. The proposal in the next section leads with Alyx for a reason. If I joined the team, the first thirty days would be spent inside that surface, not outside it. The notes below are what I'd bring to the first meeting; they aren't what I'd bring to the second.

What I observed

01
Alyx's empty state: what does it do for me on day one? From the demo and docs, Alyx is positioned as an AI agent that helps users debug their AI app. The first-load empty state is unclear from the public surface: what does Alyx know about my app before I've explained anything? Does it auto-discover from my traces? Or wait for me to type? A PM hire question, not a feedback note: what's the time-to-value loop for Alyx, and is empty-state agency the right place to invest? Tool Bypass at runtime in Alyx (the proposal's 30-day target) shifts that loop from "what do I ask Alyx" to "Alyx tells me before I ask."
02
Copilot's role versus Alyx's role isn't obvious from the marketing surface. The product pages mention both Copilot and Alyx. The line between them, Copilot as a debug companion in the product and Alyx as an agent that runs on the customer's app, is implied but not stated cleanly. A positioning question I'd want to debate internally. The shortest version of the answer ("Copilot helps you understand traces; Alyx helps your app evaluate itself") would shorten the customer onboarding conversation.
03
The Bedrock AgentCore partnership is significant and barely visible. AX shipped a partnership with AWS Bedrock AgentCore. From the product surface, this is treated as a supported integration. From the customer-decision surface, it's a major signal: Bedrock customers don't have to choose between the AWS-blessed eval path and AX. If this is the wedge for the AWS-anchored enterprise segment, treat it like the wedge. The current product copy reads as "another integration." Promote it.
04
The customer case studies are technical, not commercial. The case-study pages I read described the implementation: which models, which evals, which features. They didn't describe the commercial outcome: adoption inside the customer's company, time-to-value, second-quarter renewal signals. Add a top-line outcome metric to each case study (time-to-first-eval, percent of teams adopted, weeks-to-second-product-line). The technical depth is great. The commercial framing is what gets the next prospect to convert.

section 03 · what I'd push on

Three things, in order of conviction.

One. The trace-to-eval handoff in Phoenix should be a one-click flow from any trace, not a docs page. If the user can't get from "this trace looks suspicious" to "this eval said FAIL with this evidence" in two clicks, the eval library doesn't earn its place above the trace. This is the change with the highest first-month impact for new Phoenix users.

Two. Alyx needs a runtime grounding gate. The proposal in the next section makes this case. Tool Bypass at runtime, applied to Alyx's own action assertions, is the kind of self-eval that turns "trust me, the AI agent helped" into "here's why the AI agent's claim is grounded." The same primitive Phoenix gets in OSS lives at runtime in AX, and that composition is what justifies the AX premium tier.

Three. Customer case studies need commercial framing, not technical framing. Time-to-first-eval, percent-of-teams-adopted, weeks-to-second-product-line. The technical depth is there; the commercial signal isn't. Future prospects should read a case study and learn the buyer's outcome, not the buyer's stack.
