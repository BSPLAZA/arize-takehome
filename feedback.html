how I gathered this

Two halves of feedback, both first-hand.

Phoenix is the half I instrumented. I shipped Luke into a Phoenix Cloud workspace and the friction below comes from that session.

AX is the half I drove from inside the live workspace: Alyx prompts, the evaluator UI, the monitors page. I got six "I've opened the page for you" replies in a row. That's the AX friction.

Honesty rule: Phoenix friction is from a live install. AX friction is from a live Alyx session in the workspace. Both halves are first-hand. The Phoenix half is what I'd change as a builder. The AX half is what I'd reshape as a PM.
section 01 · phoenix

Onboarding to first trace, in one afternoon.

The friction I hit between sign-in and seeing my eval go green:

What I hit, in order

01
Client(base_url=...) needs the workspace path or every call returns 401. The Cloud quickstart shows the bare domain. Client(base_url='https://app.phoenix.arize.com') → 401. Three attempts to land on /s/bsplaza. Infer the workspace from the JWT, or return "this key is bound to workspace X; set base_url to ..." instead of a generic 401.
02
API key plumbing has five working paths and no canonical one. Client(api_key=...), PHOENIX_API_KEY, headers={'api_key':...}, Bearer header, PHOENIX_CLIENT_HEADERS. All valid, none flagged as recommended. Pick one recommended path; put it at the top of the SDK quickstart.
03
DataFrame columns aren't list-flattened the way the rest of the schema is. attributes.llm.output_messages returns a list of dicts, not flattened to attributes.llm.output_messages.0.message.content. My extractor returned empty strings until I introspected. Flatten consistently, or surface the schema in get_spans_dataframe's docstring.
04
The simplest evaluator primitive is buried. Found MatchesRegex after writing a custom regex classifier. Would have replaced 30 lines. Docs lead with the LLM judges; lightweight primitives aren't on the same page. "Before you write your own evaluator" page listing MatchesRegex, ExactMatch, PrecisionRecallFScore.
05
The trace-to-eval handoff is invisible from the UI. Trace-detail annotations panel reads "No annotation configurations for this project." No CTA, no "run a built-in eval" button. Only path is a Python SDK script. Wire the empty panel to one-click: pick eval, pick model, run on this trace. First eval should be a button, not a notebook.
06
The filter bar expects you to know the query syntax. span_kind == 'LLM' as raw input. No query builder, no column-header click-to-filter, no autocomplete. Column-header dropdowns for kind, status, latency bucket. Keep the raw query for power users.
07
URLs use base64 GraphQL IDs (UHJvamVjdDox) instead of slugs. Cosmetic, but unfriendly to share, and some browser-automation tools refuse to navigate them. Add a slug alongside the base64. Default the URL bar to the slug.
08
Annotation Configs created via REST don't show up in the trace-detail "Edit Annotations" panel. POST /v1/annotation_configs returns 200. log_span_annotations_dataframe returns 200. Annotation Summary shows the rollup. But Edit Annotations says "no configurations for this project," and POST/PUT /v1/projects/{id}/annotation_configs both return 405. Reviewers can't label without a notebook. Auto-bind workspace configs to all projects, or expose the binding endpoint over REST.
section 02 · what I tried that struggled

I tried to drive Phoenix from Claude Code as my agent. The seams showed.

Claude Code drove the instrumentation work with me supervising — the closest thing I have to a shadow-Alyx. Where it stalled is its own piece of feedback.

What worked

  • SDK ops: span queries, dataset CRUD, experiments, annotations — all driven without touching the UI
  • GraphQL mutations once the schema was discovered
  • OTLP shipping from Luke to the workspace

Where it stalled

01
Evaluator registration is a UI-only flow. No public REST or GraphQL mutation accepts prompt + model + dataset and lands an evaluator on /evaluators. I had to break out of the agent loop and click through the modal myself. Surface POST /v1/evaluators. Without it, AI-driven Phoenix workflows hit a wall exactly where the agent should keep going.
02
Schema discovery requires __schema introspection. No surfaced SDL or "what mutations exist for X" page. Each discovery cost round-trips and wrong guesses. Publish the GraphQL SDL alongside the REST OpenAPI spec on a single API reference page an agent can fetch in one read.
03
Annotation Configs created via REST don't show up in the trace UI panel. Same as section 01 friction 08. The agent writes structured data correctly; the UI still says "no configurations for this project." Auto-bind workspace configs to projects, or surface the binding as part of the create flow.
04
Modal-heavy UI is hostile to browser automation. Every modal click via browser MCP threw chrome-extension:// errors. Locked-down enterprise browsers will hit similar walls. Where workflows need a modal, also expose a same-page panel or a deep-linkable URL.
Why this matters for Arize: Alyx is the in-product agent. Anything blocking external agents from driving Phoenix end-to-end constrains what Alyx can do for the customer. Eval-registration is the highest-impact gap.
section 03 · ax

Alyx is a router, not an agent.

Six prompts in a live Alyx session in the AX workspace. Six page-handoffs in response. The pattern: Alyx interprets the intent, identifies the right page, opens it, disengages. "Alyx will help you" inside the page is a UI handoff, not a continuation. Once you click in, Alyx is gone and you're filling out the form yourself.

What I asked AlyxWhat Alyx returned
"What failure modes do you see in this project?" Asked which project. After I picked luke-75-hard: "I've opened your luke-75-hard project where Alyx will continue helping you analyze failure modes." Stopped.
"What failure modes do you see in this project?" (repeated, project context already set) "I've opened your project page where Alyx will help you identify and analyze failure modes." Listed what the page can do. Did nothing itself.
"Look at the luke-75-hard project. What failure modes do you see?" "I can see your project, but I don't currently have tools to directly query and analyze trace data." Suggested installing the arize-trace skill in Cursor or Claude Code. Routed to the project page.
"Why did experiment tool_bypass_judge_sonnet_phoenix_parity only hit 9 of 16?" "I've opened the tool_bypass_validation_v1 dataset page where Alyx will help you investigate." No analysis.
"Can you build me an evaluator that catches tool_bypass?" "I've opened the evaluator creation page where Alyx will help you build a tool bypass detection evaluator." The actual creation happens in the form I now have to fill out.
"Set up a monitor that fires when tool_bypass score drops" "I've opened the Monitors page where Alyx will help you create an alert..." Same shape.

The cost isn't one disappointing session. It's faith. After three "I've opened the page" replies, users stop asking Alyx anything substantive. By the time the agent improves, the muscle memory is "skip Alyx, just navigate." That's trust the product has to earn back when it didn't have to spend it.

What I'd change

  • Continue agentic flows in-context. When a user asks "build me an evaluator," Alyx should call evaluators.create_template_evaluator on the user's behalf, surface the result, ask follow-ups. The page-handoff is the fallback, not the default.
  • If Alyx can't act, warm handoff with context. Pre-fill the evaluator form with a draft prompt. Pre-set the monitor threshold from the conversation. Pre-filter the traces panel by the user's question. Right now the routing strips context on the way through.
Phoenix Cloud has no Alyx-equivalent. The pure existence of Alyx is an AX win. As-shipped, Alyx is doing less than the user could do themselves with the sidebar; the win only lands when Alyx earns its place above the navigation bar.

Other AX observations

01
The Bedrock AgentCore partnership reads as another integration when it could read as a wedge. AX shipped a partnership with AWS Bedrock AgentCore. From the customer-decision surface, that's a meaningful signal: Bedrock customers don't have to choose between the AWS-blessed eval path and AX. The current page treats it as a logo on a list. If the AWS-anchored enterprise segment is a real cohort, the partnership deserves a section, not a logo.
02
The customer case studies are technical, not commercial. The case-study pages describe implementation: which models, which evals, which features. They don't describe commercial outcomes: adoption inside the customer's company, time-to-value, second-quarter renewal signals. Add a top-line outcome metric to each case study (time-to-first-eval, percent of teams adopted, weeks-to-second-product-line). Technical depth is good. Commercial framing is what converts the next prospect.

section 04 · what I'd push on

Three things, in order of conviction.

  • Trace-to-eval handoff in Phoenix should be one click, not a docs page. Two clicks from "this trace looks suspicious" to "this eval said FAIL with this evidence" — or the eval library doesn't earn its place above the trace
  • Alyx needs a runtime grounding gate. Tool Bypass at runtime, applied to Alyx's own action assertions, turns "trust me, the agent helped" into "here's why the agent's claim is grounded." Same primitive in OSS, lifted into runtime in AX — the composition that justifies the AX tier
  • Customer case studies need commercial framing. Time-to-first-eval, percent-of-teams-adopted, weeks-to-second-product-line. Prospects should learn the buyer's outcome, not the buyer's stack
